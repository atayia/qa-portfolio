Test Plan

API Testing Strategy — Stripe Payments, Refunds & Error Handling

API Contract Test Plan Test Cases Bug Reports
Scope
REST API testing of Stripe's core payment lifecycle in test mode: PaymentIntents, Charges, Refunds, Auth & Security, and Input Validation. Webhooks and Connect platform features are out of scope for this audit cycle.
Testing Strategy
Black-box API testing using Postman. Approach covers:
  • Happy path — valid inputs, expected success responses
  • Negative / error path — declined cards, invalid inputs, bad auth
  • Contract verification — response structure, field presence, status codes
  • State chaining — Postman environment variables to carry IDs across requests
Environment
Base URL: https://api.stripe.com/v1
Auth: Bearer token — Stripe test secret key (sk_test_…)
API Version header: Stripe-Version: 2024-06-20
Tool: Postman (Desktop) with environment file for key and variable management
Test Data
Stripe-provided test cards only. No real card numbers used.
  • 4242 4242 4242 4242 — always succeeds (Visa)
  • 4000 0000 0000 0002 — generic decline
  • 4000 0000 0000 9995 — insufficient funds
  • 4000 0000 0000 0101 — incorrect CVC
Risk Areas
  • High: Auth failure — missing or wrong-mode API keys must be clearly rejected
  • High: Partial refund state — amount_refunded accuracy
  • Medium: Error response consistency — error.param must be present on 400s
  • Low: Metadata field passthrough — confirmed by contract but lower user impact
Out of Scope
Webhooks, Stripe Connect, Subscriptions, Invoices, Radar (fraud rules), and any live-mode transaction processing.
Exit Criteria
100% of test cases executed; all High-risk scenarios verified; all identified contract deviations documented in Bug Reports.
Deliverables
  • API Contract (RTM)
  • Test Cases
  • Bug Reports
  • Postman Collection export (coming soon)
